# GHOSTROOM PHASE 7.5: DURABLE RELAY MIGRATION ARCHITECTURE

This document details the transition from an ephemeral, Redis-only backend to a durable, multi-year message relay utilizing PostgreSQL as the System of Record.

---

## 1. STORAGE PHILOSOPHY

### PostgreSQL (System of Record)
Postgres holds the definitive, durable state of the system.
*   **Encrypted Message Envelopes**: Stored indefinitely until explicitly acknowledged/deleted or expired by global retention rules.
*   **Media Metadata**: Tracks R2 blob lifecycle states.
*   **Delivery State**: Tracks message synchronization state per recipient.

### Redis (Ephemeral State & Caching)
Redis is demoted to a high-performance ephemeral layer.
*   **Inbox Cache**: Caches recent message IDs (`inbox:{public_id}` ZSET) to serve live `inbox.fetch` requests without hitting Postgres.
*   **Presence**: Tracks active WebSocket connections and binds `socket.id -> public_id`.
*   **Auth Challenges**: Stores short-lived nonces (`challenge:{socket_id}`) for `identity.prove`.
*   **Rate Limits & Quotas**: Stores daily upload bytes and counts.

### Cloudflare R2 (Blob Storage)
*   **Encrypted Media**: Holds the encrypted bytes for images, videos, and thumbnails.
*   Lifecycles are strictly managed via Postgres metadata.

---

## 2. POSTGRESQL SCHEMA

### `messages`
*   `id` (UUID, Primary Key) - UUIDv7 generated by client.
*   `recipient_id` (VARCHAR) - Base58 Public ID of the recipient.
*   `envelope` (JSONB) - The full E2EE payload containing `k`, `n`, `c`, `s`, `v`.
*   `created_at` (TIMESTAMP) - Extracted from UUIDv7 or set on insert.
*   `expires_at` (TIMESTAMP) - Optional automatic pruning date.

### `media_metadata`
*   `id` (UUID, Primary Key)
*   `owner_id` (VARCHAR) - Base58 Public ID.
*   `size_bytes` (BIGINT)
*   `mime_type` (VARCHAR)
*   `content_hash` (VARCHAR) - SHA256.
*   `state` (VARCHAR) - 'UPLOADING', 'UPLOADED', 'REFERENCED', 'VIEWED', 'EXPIRED'.
*   `created_at` (TIMESTAMP)
*   `expires_at` (TIMESTAMP)

### `message_delivery`
*   `message_id` (UUID, Foreign Key)
*   `recipient_id` (VARCHAR)
*   `status` (VARCHAR) - 'PENDING', 'DELIVERED', 'ACKNOWLEDGED'.
*   `updated_at` (TIMESTAMP)

---

## 3. MIGRATION STRATEGY (V1/V2 COEXISTENCE)

1.  **V1 Spaces**: Will continue to use Redis `LIST` and `STRING` completely bypassing PostgreSQL. Ephemeral spaces do not require durable persistence.
2.  **V2 Inboxes**: Will dual-write.
    *   `message.send`: Insert into Postgres `messages` -> `ZADD` to Redis `inbox:{id}` cache -> Publish via Socket.IO.
    *   `inbox.fetch`: Read from Redis `inbox:{id}`. If cache miss or sync gap detected, fallback to Postgres `SELECT * FROM messages WHERE recipient_id = ? AND created_at > ?`.
    *   `message.ack`: `DELETE FROM messages WHERE id = ?` -> `ZREM inbox:{id}` -> `DEL msg:{id}`.

---

## 4. RETENTION POLICIES

While Postgres enables multi-year storage, an upper bound should be enforced to prevent infinite database growth for inactive accounts.
*   **Durable Messages**: Retained until ACK'd by client or 1 year has passed (configurable).
*   **Unreferenced Media**: Purged after 48 hours.

---

## 5. RECOVERY FLOWS

*   **Relay Restart**: Redis is wiped. Upon client connection, `inbox.fetch` queries Postgres and rehydrates the Redis `inbox:{id}` ZSET for active users.
*   **Multi-Device Restore**: A user restoring their seed phrase on a new device will fetch all unacknowledged messages directly from PostgreSQL.
