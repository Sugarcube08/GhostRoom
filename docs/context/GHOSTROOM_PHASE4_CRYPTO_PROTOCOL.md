# GHOSTROOM PHASE 4: HYBRID CRYPTO PROTOCOL

This document defines the cryptographic contract for GhostRoom V2 Direct Messaging.

---

## 1. MESSAGE ENVELOPE SCHEMA (V2)

The envelope is the unit of transport relayed by the server.

```json
{
  "id": "uuid_v7",
  "k": "base64(encrypted_message_key)",
  "n": "base64(nonce)",
  "c": "base64(ciphertext)",
  "s": "base64(signature)",
  "v": 2
}
```

*   **id**: UUID v7 (Time-sortable).
*   **k** (Key): The 32-byte Message Key, encrypted using `crypto_box_seal` for the recipient's X25519 Public Key.
*   **n** (Nonce): 24-byte random nonce for the symmetric encryption.
*   **c** (Ciphertext): The payload encrypted with XChaCha20-Poly1305 using the Message Key.
*   **s** (Signature): Ed25519 signature of the header + encrypted material.
*   **v**: Protocol version (Fixed at `2`).

---

## 2. SIGNATURE SCOPE

To prevent tampering and relay-side substitution, the signature covers the entire mutable and immutable context of the message.

**Signed Material**:
`id` + `k` + `n` + `c`

The signature is generated by the sender's **Ed25519 Secret Key**.

---

## 3. CRYPTOGRAPHIC WORKFLOW

### Sender Path:
1.  **Generate Message Key**: Create a random 32-byte symmetric key.
2.  **Encrypt Payload**: Use XChaCha20-Poly1305 with the Message Key and a fresh random 24-byte nonce to encrypt the message JSON.
3.  **Wrap Key**: Use `crypto_box_seal` to encrypt the Message Key for the Recipient's X25519 Public Key.
4.  **Generate ID**: Generate a UUID v7.
5.  **Sign**: Sign the concatenation of `id + k + n + c` using Ed25519.
6.  **Transmit**: Send the resulting JSON envelope to the Relay.

### Recipient Path:
1.  **Verify Signature**: Check `s` against the sender's known Ed25519 Public Key using `id + k + n + c`.
2.  **Unwrap Key**: Use `crypto_box_seal_open` with own X25519 Secret Key to recover the Message Key from `k`.
3.  **Decrypt Payload**: Use XChaCha20-Poly1305 with the recovered Message Key and nonce `n` to recover the plaintext.

---

## 4. REPLAY & DUPLICATE PROTECTION

*   **UUID v7 Enforcement**: Clients must reject messages with IDs that are smaller (older) than the `last_processed_timestamp` unless explicitly performing a full sync.
*   **Deduplication**: Clients maintain a local Hive box of `processed_message_ids`. Any incoming `id` already in this box is `ACK`'d to the server but discarded from the UI.

---

## 5. NONCE STRATEGY

*   **Length**: 24 bytes (XChaCha20-Poly1305 standard).
*   **Entropy**: Must use a cryptographically secure random number generator (CSPRNG).
*   **Storage**: Included in the plaintext envelope.

---

## 6. KEY USAGE SUMMARY

| Algorithm | Usage |
| :--- | :--- |
| **Ed25519** | Identity Proof (Auth) and Message Signing (Integrity). |
| **X25519** | Message Key Wrapping (Privacy). |
| **XChaCha20-Poly1305** | Bulk Message/Media Encryption (Symmetric). |

---

## 7. FAILURE STATES

| Failure | Client Action |
| :--- | :--- |
| **Invalid Signature** | Discard, Send `ACK` (Clear poison pill), Log security warning. |
| **Decryption Failure** | Discard, Send `ACK`, Log error. |
| **Expired Message** | If server delivers message older than local threshold, discard and `ACK`. |
| **Duplicate ID** | Discard, Send `ACK`. |

---

## 8. BACKWARD COMPATIBILITY

*   **V1 Spaces**: Continue using `secretBox` (Symmetric) with the `roomId` derived key. The envelope lacks the `k` field and uses `v: 1`.
*   **V2 DMs**: Always use the Hybrid model described above with `v: 2`.
